## Ambiguous CampaignTemplate fields encountered and how they were resolved

### 1. `budget_total_usd: 0.0` at YAML top level

**Not** a field on `core.campaign.CampaignCreate` or `ai_orchestrator_client.CampaignCreate` (verified by reading both Pydantic models). The SDK's `model_config = ConfigDict(extra="ignore")` silently drops it on `CampaignCreate(**data)`. Even if it were threaded through, `core/budget.py` treats `<=0` as "no limit / unlimited":

```python
# core/budget.py
if budget_total_usd is None or budget_total_usd <= 0:
    return None  # percentage_used = None, no cap enforcement
```

**Resolution per operator choice in plan-time discussion: keep the line as the brief wrote it.** The line documents intent — *these campaigns are expected to cost zero USD because they route through local Ollama* — but is currently a no-op. Documented in `RUNBOOK.md` § "budget_total_usd current behavior". If a real cap becomes necessary (paid-provider routing), file an issue against ai-orchestrator to add `budget_total_usd` to `CampaignCreate.model_validate`.

### 2. Spec's flat directory layout vs. namespace-collision

The Stage-3 brief writes a flat repo layout: `evidence/`, `geometry/`, `cfd/`, etc. at repo root. This would collide with the orchestrator's own `/opt/ai-orchestrator/evidence/` package whenever both end up on the same `sys.path` (which they will, in Stage 5/6 when the orchestrator needs to `pip install -e /opt/aero-research-platform` for pluggy entry-point discovery).

**Resolution: nest all Python under a single namespace package** `aero_research_platform/` with submodules `evidence/`, `geometry/`, `meshing/`, `cfd/`, `surrogates/`, `optimization/`, `llm/`. The repo's `campaigns/`, `sky/`, `scripts/`, `tests/`, `notebooks/`, `data/`, `results/` stay at repo root (they aren't Python packages). Entry points in `pyproject.toml` target `aero_research_platform.evidence.*`. This is the only departure from the spec's literal directory layout and is required for forward compatibility with the pluggy plugin host.

### 3. SDK pin "must NOT be PyPI" per audit B.2

Stale audit — see § "SDK pin disposition" above. Resolution: PyPI range pin per operator scalability preference.

### 4. Repo orchestrator-side documentation links

Spec audit B.1: orchestrator's `VISION.md` does not exist. **Resolution:** the aero repo's docs link to the orchestrator's `CLAUDE.md` and `CONSUMERS.md` instead. Internal `[VISION.md](VISION.md)` references in this repo are to the aero repo's own `VISION.md`, which DOES exist (commit 2).
